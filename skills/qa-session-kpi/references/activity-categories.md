# QA 활동 유형 분류 기준

Claude Code 세션에서 사용자 메시지를 5개 유형으로 분류할 때 이 기준을 따른다.

---

## issue_found — 이슈·버그 발견

사용자가 문제를 처음으로 인식하고 보고하는 메시지. 이미 알려진 이슈의 재언급은 제외.

**포함되는 패턴:**
- "버그 발견했어", "이거 안 돼", "에러 나", "깨져", "이상한데"
- "이 케이스 누락된 것 같아", "여기 예외 처리 없어"
- 스택트레이스·로그를 붙여넣으며 문제를 처음 제기할 때

**제외:**
- 이미 보고한 이슈를 다시 언급 → `fix_instruction` 또는 `verification`
- "이 코드 뭐가 문제야?" (원인 궁금증) → `analysis`

---

## fix_instruction — 수정·변경 지시

사용자가 AI에게 코드·동작·설정을 바꾸도록 요청하는 메시지.

**포함되는 패턴:**
- "고쳐줘", "수정해줘", "바꿔줘", "이렇게 해줘", "추가해줘"
- "이 조건 추가해", "null 체크 넣어줘"

**제외:**
- 수정 결과를 확인하는 것 → `verification`

---

## verification — 검증·확인

수정 또는 구현 후 결과를 직접 확인하거나 승인하는 메시지.

**포함되는 패턴:**
- "됐어", "OK", "통과", "확인했어", "테스트 해봤는데 됨"
- "이 케이스는 통과, 근데 저 케이스는 아직 안 돼"
- "실기기에서 확인함", "화면 캡처 보내줄게"

**제외:**
- 버그를 발견하는 것 → `issue_found`
- 원인을 찾는 것 → `analysis`

---

## test_case — 테스트 케이스·시나리오 작성

테스트 케이스, 체크리스트, 시나리오를 만들거나 정의하는 메시지.

**포함되는 패턴:**
- "테스트 케이스 만들어줘", "어떤 경우를 테스트해야 해?"
- "시나리오 작성해줘", "경계값 테스트 추가해줘"
- 테스트 목록을 직접 작성하거나 AI에게 작성 요청

**제외:**
- 단순히 테스트를 실행하는 것 → `verification`

---

## analysis — 원인 분석·조사

문제의 원인을 파악하거나 동작을 이해하려는 메시지.

**포함되는 패턴:**
- "왜 이래?", "원인이 뭐야?", "로그 분석해줘"
- "이 코드가 어떻게 동작하는 거야?", "재현 조건이 뭔지 파악해줘"
- 코드 흐름·의존성 추적 요청

**제외:**
- 문제를 처음 발견하는 것 → `issue_found`

---

## 판단이 어려운 케이스

| 메시지 예시 | 유형 | 이유 |
|-------------|------|------|
| "이거 왜 이래? 고쳐줘" | `fix_instruction` | 수정 의도가 지배적 |
| "확인했는데 이 케이스는 아직 안 돼" | `issue_found` | 새로운 미해결 이슈 제기 |
| "로그 보니까 null이 들어와. null 체크 추가해줘" | `fix_instruction` | 분석 결과가 있지만 최종 의도는 수정 지시 |
| "테스트 돌려봤더니 됐어" | `verification` | 테스트 실행 + 확인 |
